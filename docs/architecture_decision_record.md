# Architecture Decision Record (ADR) — единая подсудность

Версия: 1.0 (черновик для согласования)  
Дата: 2025-03-26

Цель: минимизировать риски, ускорить внедрение и обеспечить долгосрочную поддержку за счёт фиксации «источника правды», единых точек входа и схемы данных.

---

## Фаза 0. Определение «источника правды» (1–2 дня)

**Цель:** зафиксировать целевое состояние системы.

**Действия:**

1. Этот документ — основа для ADR со следующими решениями:
   - **Главный клиент:** HTTP API `POST /api/v1/find-court` (`unified_jurisdiction`) — **целевая** единая точка входа для всех потребителей (бот, batch, скрипты могут вызывать HTTP или in-process SDK).
   - **Формат ответа:** единый контракт JSON (ниже). Текущая реализация близка к нему; расширения (`confidence_score`, `metrics`) — целевое состояние после фазы 1.
   - **Источник пространственных данных:** одна таблица `court_districts` в **PostGIS** (после фазы 3); до миграции допускается SQLite как временный слой с явной пометкой в документации.
   - **Стратегия fallback:** полигон (`ST_Contains` / аналог) → район + справочник судов → ближайший суд в радиусе → флаг ручной проверки.
   - **Политика секретов:** только `.env` локально и secret store в проде; `.env` не в git.

2. Согласование с командой: 1–2 встречи, список подписантов в конце документа.

**Критерий готовности:** документ утверждён (подписи / протокол).

### Целевой контракт ответа API (JSON)

```json
{
  "success": true,
  "court": {
    "court_name": "Мировой судья судебного участка № …",
    "court_code": "опционально после фазы 3",
    "address": "",
    "section_num": 0,
    "region": "",
    "source": "unified_B_dadata"
  },
  "unified_address": {
    "raw": "",
    "normalized": "г. Москва, ул. Примерная, д. 1",
    "region": null,
    "district": null
  },
  "resolution_steps": ["B:dadata", "A:district_db"],
  "needs_manual_review": false,
  "spatial_override": false,
  "confidence_score": 0.95,
  "error": null,
  "metrics": {
    "dadata_ms": 120,
    "geocode_ms": 80,
    "total_ms": 450
  }
}
```

**Текущее состояние (реализовано в коде):** `unified_address` — объект; в ответ API добавлены **`confidence_score`** (эвристика по `source`) и **`metrics.total_ms`**; batch по умолчанию идёт через `UnifiedJurisdictionClient` (см. [entry_points.md](entry_points.md)).

## Фаза 1. Быстрый выигрыш: надёжность и тесты (1–3 недели)

**Цель:** повысить доверие к системе и упростить отладку.

**Действия:**

- **Ошибки:** сократить «глухие» `except Exception: pass` в цепочке геокодирования/матчинга; логировать тип этапа, класс исключения, HTTP-статус (без утечки ключей и полных тел ответов).
- **Логирование:** единая схема полей: `resolution_steps`, `yandex_geo=GEOCODER<-ENV`, источник DaData, `success`/`fail`.
- **Метрики (по желанию):** счётчики Prometheus (`dadata_requests_total`, `geocoder_errors_total`, длительности гистограммами) — если инфраструктура уже есть; иначе начать с структурных логов и таймингов в JSON ответа.
- **Тесты:**
  - Обновить `tests/test_apis_connectivity.py`: согласовать проверку ключей с `court_locator.config` (не полагаться только на `YANDEX_API_KEY` для геокодера).
  - Дымовые тесты: 5–10 адресов/координат с проверкой **набора** `resolution_steps` и `success`, без жёсткой привязки к названию суда при смене данных.
  - CI: запуск с предсказуемым `PYTHONPATH` (корень репозитория).

**Критерий готовности:** по логу за одну минуту видно этап сбоя (геокод, БД, полигон, радиус).

## Фаза 2. Унификация точек входа (2–4 недели)

**Цель:** убрать дублирование логики.

**Действия:**

1. Перевести `batch_processing`, бота, скрипты на `UnifiedJurisdictionClient` (local или HTTP).
2. `parser/jurisdiction.py`: вариант A — тонкая обёртка над unified; вариант B — `legacy` + дата отключения в документе.
3. `court_locator` REST: deprecation в пользу `POST /api/v1/find-court` или тонкий прокси.
4. `docs/entry_points.md` — таблица «модуль → куда ходит».

**Критерий готовности:** нет второй полноценной бизнес-логики подсудности вне unified-слоя (кроме явного legacy с датой).

---

## Фаза 3. Единая модель данных (3–6 недели)

**Цель:** убрать расхождение схем PostGIS / SQLite / сервисов.

**Действия:**

1. Утвердить одну схему `court_districts` (имя геополя, CRS 4326, `valid_from` / `valid_to`, при необходимости `source`, связь с справочником суда).
2. Миграция: `courts.sqlite` → Postgres; полигоны из SQLite/GeoJSON/NextGIS → та же БД.
3. Единый `PG_DSN` для приложений; зафиксировать одну SQL-миграцию как канон (свести `migrations/001_...`, `jurisdiction_service`, адаптеры).
4. Обновить `postgis_adapter.py` (и аналоги) под канон.
5. Удалить или пометить **deprecated** дублирующие миграции после переноса.

**Критерий готовности:** один документ «схема БД», один способ пространственного запроса для продакшена.

*Черновик DDL в вашем плане (таблица с `boundary`, `court_code`, …) — согласовать с фактическими полями справочника судов и импортом из текущих файлов.*

## Фаза 4. Безопасные импорты и CI (3–5 дней)

**Цель:** исключить коллизию со stdlib `parser`.

**Действия:**

1. Заменить хрупкие `from parser.dadata_api` на загрузку через корень проекта / пакет с явным именем (например, будущий `parsersud`).
2. CI: тест с минимальным `PYTHONPATH`.
3. Рассмотреть `parser/__init__.py` или переименование пакета — по итогам оценки объёма правок.
4. Проверка репозитория на случайные секреты (pre-commit или скрипт).

**Критерий готовности:** нет сценария, где импортируется stdlib `parser` вместо проекта.

## Фаза 5. Документация и операционка (ongoing)

**Цель:** поддерживаемость и актуальность данных.

**Действия:**

1. Синхронизировать `apis_and_keys.md`, `what_is_missing_for_search` с реальными путями (Yandex+БД без DaData и т.д.).
2. Чек-лист сезонного обновления: суды, полигоны, ключи API.
3. Зафиксировать политику Redis: fail-open vs fail-closed по сервисам.
4. Мониторинг до/после миграции БД (латентность, ошибки геокодера).

**Критерий готовности:** новый разработчик по докам понимает входы, секреты и процесс обновления данных.

## Итог

Объединённый план:

- **Закрывает выявленные проблемы:** разрозненные схемы, хрупкие импорты, скрытые ошибки, устаревшие подсказки в доке.
- **Даёт быстрый выигрыш:** уже после **фазы 1** снимается основная боль отладки (видно этап и причину сбоя) и повышается доверие к регрессии за счёт дымовых тестов.
- **Снижает стратегический риск:** **фаза 3** убирает главный источник расхождений (несколько определений `court_districts` и разные имена геомполей).
- **Упрощает сопровождение:** **фазы 2, 4, 5** направлены на одну точку входа, предсказуемые импорты и актуальную операционную документацию.
